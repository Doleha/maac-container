# MAAC Container

**Phase 2 client relay for MAAC certification.**

This container runs inside your environment during the Phase 2 stage of a MAAC (Muslim-American AI Certification) engagement. It connects to the MaacVerify server, receives your submitted operational scenarios one at a time, routes each one through your AI model, and returns the encrypted responses for MAAC assessment.

The result is a certification report that tells you whether the cognitive profile established in Phase 1 — measured on synthetic benchmarks — holds up under the real-world conditions in which you actually deploy your model.

---

## The Certification Pipeline

MAAC certification runs in two phases. The container is the bridge between them.

### Phase 1 — Synthetic Benchmark

MaacVerify runs your model through a structured set of synthetic scenarios designed to assess nine cognitive dimensions. The results produce a **cognitive architecture profile**: a per-dimension characterization of how the model reasons, including floor thresholds (θ) and equivalence bounds (Δ) that define what "same model performance" looks like statistically.

Phase 1 is locked once it passes quality diagnostics. After locking, the profile and evaluator version are immutable — no parameters can be adjusted before Phase 2.

### Phase 2 — Live Operational Assessment

You submit your own real-world scenarios to MaacVerify: actual decision tasks from the context in which you deploy this model, de-identified as required by your engagement agreement. These are not generated by MaacVerify — they are yours.

The container routes those scenarios through your model inside your infrastructure, returning the responses encrypted. MaacVerify assesses the responses against the same nine MAAC dimensions using the same evaluator version locked in Phase 1, then runs a statistical comparison against the Phase 1 profile.

This answers the question that Phase 1 cannot: **does the model perform in production the way it performed on the benchmark?**

### The Certification Outcome

The report compares your Phase 2 operational results against the Phase 1 profile and produces one of four outcomes:

| Outcome | Meaning |
|---|---|
| **Profile Confirmed** | Phase 1 characterization is valid under your real operational conditions. The model you benchmarked is the model you deployed. |
| **Profile Confirmed with Operational Shift** | The Phase 1 profile holds at the distributional level, but specific dimensions show measurable mean shifts under production conditions. Shift details are disclosed per dimension. |
| **Profile Departure Detected** | Real-world performance fell below the Phase 1 floor on one or more dimensions. The Phase 1 characterization does not hold for this deployment context. |
| **Characterization Invalid for Deployment Context** | Both distributional floor and profile mean departed. The Phase 1 profile cannot be relied upon for this operational context. |

The report includes the full statistical record: per-dimension results with confidence intervals and effect sizes, tier distributions, the nine-dimensional radar comparing Phase 1 vs. Phase 2 profiles, and — when shifts are detected — a shift registry with root cause classification and monitoring recommendations.

## Before Running This Container

The following must be in place:

1. **Phase 1 is complete and locked.** Your MaacVerify operator will confirm this. The container enforces this at connection time — the session will be rejected if Phase 1 is not locked.

2. **Your scenarios have been submitted.** You upload your operational scenarios to MaacVerify via the intake API (`POST /api/assessment/scenarios/import`) before the assessment session begins. The container runs the scenarios that are already on file — it does not accept new submissions during the session.

3. **You have a CLIENT_API_KEY.** Your operator issues this after creating the assessment session. It is single-use and tied to your specific engagement. Do not share it.

4. **Your AI model is reachable from this container.** The model endpoint must be accessible from wherever you run Docker. The container makes outbound HTTP requests to your model and outbound WebSocket connections to the MaacVerify server.

## Running the Container

### docker run

```bash
docker run --rm \
  -e MAAC_SERVER_URL="wss://provided-by-your-operator" \
  -e CLIENT_API_KEY="provided-by-your-operator" \
  -e CLIENT_AI_ENDPOINT="http://your-model-endpoint/v1/chat/completions" \
  -e CLIENT_AI_FORMAT="openai" \
  -e CLIENT_AI_AUTH_TOKEN="your-model-bearer-token" \
  -e CLIENT_AI_MODEL="your-model-name" \
  -p 8080:8080 \
  maac-container
```

### Docker Compose

Create a `.env` file (keep it private — it contains your session key):

```env
MAAC_SERVER_URL=wss://provided-by-your-operator
CLIENT_API_KEY=provided-by-your-operator
CLIENT_AI_ENDPOINT=http://your-model-endpoint/v1/chat/completions
CLIENT_AI_FORMAT=openai
CLIENT_AI_AUTH_TOKEN=your-model-bearer-token
CLIENT_AI_MODEL=your-model-name
```

```yaml
# docker-compose.yml
services:
  maac-container:
    image: maac-container
    env_file: .env
    ports:
      - "8080:8080"
    restart: "no"
```

```bash
docker compose up
```

Set `restart: "no"`. The container exits when the session completes, and the `CLIENT_API_KEY` cannot be reused.

## Environment Variables

### Required

| Variable | Description |
|---|---|
| `MAAC_SERVER_URL` | WebSocket URL provided by your operator. Starts with `wss://` (or `ws://` for local testing). |
| `CLIENT_API_KEY` | Single-use session key issued by your operator for this assessment. |
| `CLIENT_AI_ENDPOINT` | Full URL of your AI model's HTTP endpoint. Must accept `POST` with a JSON body. |
| `CLIENT_AI_FORMAT` | Request format to use when calling your model. One of: `openai`, `anthropic`, `custom`. |
| `CLIENT_AI_AUTH_TOKEN` | Bearer token for authenticating requests to your model endpoint. |

### Optional

| Variable | Default | Description |
|---|---|---|
| `CLIENT_AI_MODEL` | _(not sent)_ | Model identifier included in the request body. Required by most hosted APIs. |
| `CLIENT_AI_TIMEOUT_MS` | `120000` | Per-request timeout in milliseconds. Increase for slower models. Minimum `1000`. |
| `CLIENT_AI_MAX_RETRIES` | `3` | Retry attempts on model errors, with backoff: 2s, 4s, 8s. Set to `0` to disable. |
| `CUSTOM_BODY_TEMPLATE` | _(none)_ | Required when `CLIENT_AI_FORMAT=custom`. See below. |

## Supported Model Formats

### `openai`

Compatible with OpenAI, Azure OpenAI, and any OpenAI-compatible endpoint (vLLM, llama.cpp, Ollama, etc.).

```
CLIENT_AI_FORMAT=openai
CLIENT_AI_ENDPOINT=https://api.openai.com/v1/chat/completions
CLIENT_AI_MODEL=gpt-4o
```

### `anthropic`

Compatible with the Anthropic Messages API.

```
CLIENT_AI_FORMAT=anthropic
CLIENT_AI_ENDPOINT=https://api.anthropic.com/v1/messages
CLIENT_AI_MODEL=claude-sonnet-4-6
```

### `custom`

For endpoints that expect a non-standard request body. Provide a JSON template with `{{placeholder}}` fields. The template must produce valid JSON after substitution.

Available placeholders:

| Placeholder | Content |
|---|---|
| `{{taskTitle}}` | Short title of the task |
| `{{taskDescription}}` | Full task description |
| `{{businessContext}}` | Background context for the scenario |
| `{{scenarioRequirements}}` | Requirements the response must address |
| `{{dataElements}}` | Structured data provided with the scenario |

Example:

```
CLIENT_AI_FORMAT=custom
CUSTOM_BODY_TEMPLATE={"prompt":"{{taskTitle}}\n\n{{taskDescription}}\n\nContext: {{businessContext}}","max_new_tokens":2048}
```

Open `http://localhost:8080` in a browser while the container is running to see a live progress dashboard. It updates every 2 seconds.

For programmatic monitoring:

```bash
curl http://localhost:8080/health
```

```json
{
  "status": "running",
  "sessionId": "a1b2c3d4-...",
  "requestCount": 60,
  "scenariosComplete": 34,
  "retriesTotal": 1
}
```

| `status` | Meaning |
|---|---|
| `ready` | Connected, session handshake in progress |
| `running` | Actively processing scenarios |
| `complete` | All scenarios finished — responses delivered to MaacVerify |
| `error` | Session ended due to an error |

## What Happens During the Session

1. The container connects to MaacVerify and authenticates with your `CLIENT_API_KEY`.
2. MaacVerify sends a session key, RSA-encrypted with a public key generated by the container. Only this container instance can decrypt it.
3. Each scenario from your submission arrives as an AES-256-GCM encrypted payload. The container decrypts it locally, formats a prompt, and sends it to your model.
4. The model's response is re-encrypted with the session key before being sent back. MaacVerify receives the ciphertext — not the plaintext response.
5. This repeats until all scenarios are complete.
6. MaacVerify runs MAAC assessment on the responses and produces the certification report. Your operator will share the report with you.

Your model responses are encrypted before leaving your network. MaacVerify decrypts them server-side only for assessment. No plaintext response data is stored after assessment is complete.

---

## If the Connection Drops

The container retries up to 3 times with a 5-second delay. If it reconnects within 5 minutes, the session resumes from the last completed scenario — nothing is re-run. If the 5-minute window expires, the session aborts. Contact your operator to create a new session.

## Common Errors

**`[FATAL] Missing required environment variables`**
One or more of the five required variables is missing or empty.

**`AUTH_REJECTED: Invalid credentials`**
The `CLIENT_API_KEY` is wrong, has already been used in a completed session, or the session was cancelled. Contact your operator.

**`AUTH_REJECTED: Session is closed`**
This session ran to completion or was manually closed. A new session is needed.

**`AUTH_REJECTED: Duplicate connection`**
Another container instance is already connected with this key. Stop the other instance first.

**`[relay] Client AI returned 401`**
`CLIENT_AI_AUTH_TOKEN` is invalid. Verify the token against your model provider.

**`[relay] Client AI returned 404`**
`CLIENT_AI_ENDPOINT` is unreachable or incorrect. Confirm the model service is running.

**`scenariosComplete` not increasing**
The container is dispatching scenarios but your model is returning errors. Check the container logs for `[relay]` lines to see the error code.

## Security

- The container's RSA private key exists only in memory and is zeroed immediately after use. Never written to disk or logged.
- The AES-256-GCM session key is generated fresh per connection and lives only in memory.
- No scenario content, prompts, or model responses are written to disk at any point.
- `CLIENT_API_KEY` is never logged.
- The container runs as a non-root user with no volumes.
